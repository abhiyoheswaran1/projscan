# projscan 2.0 Platform Design

Date: 2026-05-18

## Summary

projscan 2.0.0 should turn the 1.10/1.11 plugin preview into a stable local
platform contract and use the major version boundary to clean public surfaces
that were intentionally deferred during 1.x. The release should be boring for
users who call the MCP tools normally, and explicit for users who consume the
CLI JSON or npm TypeScript API.

This is an implementation program, not release approval. The final tag, npm
publish, GitHub Release, MCP Registry publish, and website update must wait for
explicit human approval after all implementation, bug hunting, verification,
and release prep are complete.

## Product Goal

Make projscan the stable local code-intelligence substrate that other tools can
extend. A plugin author should be able to build an analyzer or CLI reporter
from docs alone, validate it locally, and expect the contract to hold across
2.x.

## Non-Goals

- No remote plugin registry.
- No plugin installation command.
- No LLM execution inside projscan.
- No SaaS, dashboard, IDE extension, or hosted service.
- No code-mod/plugin write API in 2.0.0.
- No release/tag/publish without explicit approval after release prep.

## Phase Gates

Each phase must complete in this order:

1. Implement the scoped change.
2. Run focused tests for that change.
3. Run a bug hunt specific to the phase.
4. Fix discovered bugs.
5. Run verification for the phase.
6. Commit the phase.
7. Move to the next phase only after the worktree is clean.

After all phases, run a full bug hunt across the whole 2.0 surface, fix any
issues, run full verification, complete release prep, and stop before tagging.

## Phase 0: Release Hygiene

Fix known release-pipeline risks before changing product behavior:

- Keep `.github/mcp-registry/server.json` valid under live registry limits.
- Add or preserve tests for descriptor version alignment and description length.
- Address the GitHub Actions Node 20 deprecation warning without weakening the
  release workflow.

Acceptance:

- `npm test -- tests/mcp/registryDescriptor.test.ts`
- Release workflow still validates package and MCP registry versions before
  publish.

## Phase 1: 2.0 Contract Audit

Create a committed contract audit that lists:

- stable MCP tools and input schemas
- CLI commands, documented flags, and exit codes
- CLI JSON top-level keys
- `.projscanrc` keys
- npm package exports
- intentional 2.0 breaking changes
- compatibility behavior that remains for one 2.x cycle

Acceptance:

- `docs/2.0-MIGRATION.md` exists with a practical migration guide.
- `docs/STABILITY.md` is updated for the 2.x contract.
- No code behavior changes in this phase except release workflow hygiene.

## Phase 2: Plugin API RC

Promote the local plugin contract from preview-shaped to 2.0-ready:

- Add a machine-readable manifest JSON schema for `.projscan-plugin.json`.
- Keep supported kinds to `analyzer` and `reporter`.
- Freeze analyzer `check(rootPath, files)` for 2.0.
- Freeze reporter `render(context)` for `doctor`, `analyze`, and `ci`.
- Add example analyzer and reporter plugins under docs examples.
- Test the examples so docs cannot drift.
- Keep execution local-only and opt-in through the existing preview flag until
  the final 2.0 release decision.

Acceptance:

- Plugin schema validates all documented examples.
- `projscan plugin validate` catches invalid manifests with structured
  diagnostics.
- Analyzer plugins still enter `doctor`, `analyze`, and `ci`.
- Reporter plugins still render CLI output only.

## Phase 3: Deprecated Extractor Removal

Remove the JS/TS regex import/export extractors deprecated in 0.17:

- Replace remaining CLI and MCP explain helpers with graph-backed inspection.
- Remove `extractImports` and `extractExports` exports from `fileInspector`.
- Preserve file explanation behavior for supported languages by using
  `buildCodeGraph`.
- Return empty imports/exports only when a file cannot be parsed.

Acceptance:

- No remaining references to `extractImports` or `extractExports`.
- Existing file/explain tests pass or are updated to graph-backed behavior.
- Non-JS language explanations use AST graph data.

## Phase 4: Language Extensibility

Make language identity plugin-extensible without destabilizing built-ins:

- Keep a documented `BuiltinLanguageId` union for built-in adapters.
- Widen `LanguageId` to accept plugin-provided language ids.
- Keep adapter registry behavior deterministic.
- Do not ship language plugins in 2.0.0 unless the core contract requires it.

Acceptance:

- Built-in adapter tests continue to pass.
- Type-level tests or compile checks prove custom language ids can type-check.
- `detectLanguages` and graph code keep their current runtime behavior.

## Phase 5: Versioned CLI JSON

Normalize machine-readable CLI JSON around explicit schema metadata:

- Add `schemaVersion: 2` to CLI JSON payloads emitted by built-in reporters.
- Keep existing data keys present during the 2.0 transition.
- Document that programmatic consumers should branch on `schemaVersion`.
- Keep MCP output shapes stable unless a specific 2.0 break is documented.

Acceptance:

- JSON reporter tests assert `schemaVersion: 2`.
- Command-specific top-level data remains present.
- Migration guide explains the change.

## Phase 6: HTML/Theming Through Reporter Plugins

Do not grow core HTML theming. Clarify that custom presentation belongs in
reporter plugins:

- Document reporter plugins as the theming path.
- Keep existing HTML reporter behavior working.
- Avoid adding core white-label flags.

Acceptance:

- README, guide, and plugin authoring docs agree on this boundary.
- HTML reporter tests still pass.

## Phase 7: 2.0 Release Prep Without Release

Prepare the release artifacts, but do not tag or publish:

- Bump version fields to `2.0.0` only after all implementation phases pass.
- Update `CHANGELOG.md` with `2.0.0`.
- Update `.github/mcp-registry/server.json` to `2.0.0`.
- Update roadmap, README, guide, stability docs, migration guide, and package
  description.
- Regenerate build outputs and tool manifest.
- Support prerelease workflows only if needed for RC tags; final release still
  waits for approval.

Acceptance:

- `npm run build`
- `npm run lint`
- `npm test`
- `npm run check:stability`
- `node ./dist/cli/index.js doctor --format json --quiet`
- `npm pack --dry-run`
- live MCP registry validation if network/auth allows

## Final Bug Hunt

Before asking for release approval:

- Run full tests.
- Run focused plugin analyzer/reporter fixture tests.
- Run CLI smoke tests for `doctor`, `analyze`, `ci`, `file`, `explain`, and
  `plugin validate`.
- Run `projscan doctor` against projscan itself.
- Run `npm pack --dry-run` and inspect package contents.
- Search for stale `1.11`, preview-only claims, missing migration notes, and
  deprecated extractor references.
- Fix any issue found, then rerun the relevant focused test and full gate.

## Stop Condition

When release prep is complete, stop with:

- branch status
- commit list
- verification evidence
- known residual risks
- exact command the user should approve next

Do not create `v2.0.0`, push the tag, publish npm, publish MCP Registry, or ask
the website to update until the user gives explicit release approval.
