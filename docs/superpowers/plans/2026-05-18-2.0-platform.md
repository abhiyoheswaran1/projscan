# projscan 2.0 Platform Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stabilize the local plugin platform and prepare projscan 2.0.0 without tagging or publishing until explicit human approval.

**Architecture:** Keep behavior changes in narrow modules: release validation in workflow/tests, plugin contract in `src/core/plugins.ts` plus docs schema/examples, file explanation in graph-backed inspection helpers, language identity in `src/core/languages/LanguageAdapter.ts`, and CLI JSON metadata in `src/reporters/jsonReporter.ts`. Each phase ends with focused tests, a bug hunt, fixes, verification, and a commit.

**Tech Stack:** TypeScript, Node.js ESM, Commander CLI, MCP tools, Vitest, GitHub Actions, npm release workflow, MCP Registry descriptor.

## Task 1: Contract Audit and Migration Guide

**Files:**

- Create: `docs/2.0-MIGRATION.md`
- Modify: `docs/STABILITY.md`

- [ ] **Step 1: Add migration guide**

Create `docs/2.0-MIGRATION.md` with these sections:

```md
# Migrating to projscan 2.0

projscan 2.0 stabilizes the local plugin platform and uses the major version
boundary to clean up deferred 1.x surfaces.

## Breaking Changes

### CLI JSON has `schemaVersion: 2`

Every built-in CLI JSON reporter now includes `schemaVersion: 2` at the top
level. Existing command data keys remain present.

### Deprecated regex extractors are removed

The deprecated `extractImports` and `extractExports` helpers are removed from
the npm public API. Use `buildCodeGraph`, `importsOf`, `exportsOf`, or
`inspectFile` instead.

### Language ids are extensible

Built-in language ids remain documented, but `LanguageId` is no longer a closed
union. Consumers that need only built-ins should use `BuiltinLanguageId`.

## Plugin Contract

Plugin manifests use `schemaVersion: 1` in 2.0. Analyzer plugins export
`check(rootPath, files)`. Reporter plugins export `render(context)`.

## Compatibility Notes

MCP tool names and input schemas remain stable. The major-version changes are
limited to documented CLI JSON metadata, npm TypeScript API cleanup, and plugin
contract stabilization.
```

- [ ] **Step 2: Update stability docs**

In `docs/STABILITY.md`, add:

```md
### Plugin API

Starting in 2.0, `.projscan-plugins/*.projscan-plugin.json` manifests with
`schemaVersion: 1` are stable for analyzer and reporter plugins. New optional
manifest fields may be added in 2.x; existing required fields keep their names
and types.
```

Also update the npm public API section to mention `BuiltinLanguageId` for
